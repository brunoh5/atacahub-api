# Requisitos Não Funcionais (RNF)

# Módulo 02 — API REST

# RNF021 → RNF040 — API REST

## RNF021 — Arquitetura RESTful

### Requisito

A API deverá seguir os princípios da arquitetura REST.

### Justificativa

Promove padronização, interoperabilidade e facilidade de integração.

---

## RNF022 — Comunicação em JSON

### Requisito

A API deverá utilizar JSON como formato padrão para troca de dados.

### Justificativa

JSON é amplamente suportado e reduz a complexidade de integração.

---

## RNF023 — Versionamento - ✅ Concluído

### Requisito

Toda API deverá ser versionada através da URL utilizando o padrão `/api/v{versão}`.

### Justificativa

Permite evolução da API sem quebrar integrações existentes.

---

## RNF024 — UTF-8

### Requisito

Todas as requisições e respostas deverão utilizar codificação UTF-8.

### Justificativa

Garante compatibilidade internacional.

---

## RNF025 — HTTPS Obrigatório

### Requisito

A API deverá aceitar apenas conexões utilizando HTTPS em ambiente de produção.

### Justificativa

Protege os dados transmitidos entre cliente e servidor.

---

## RNF026 — Códigos HTTP

### Requisito

Os endpoints deverão utilizar códigos HTTP compatíveis com a operação executada.

### Justificativa

Facilita o entendimento das respostas por clientes e ferramentas.

---

## RNF027 — Padronização de Respostas

### Requisito

Todas as respostas deverão seguir um contrato padronizado.

### Justificativa

Facilita o consumo da API e reduz inconsistências.

---

## RNF028 — Padronização de Erros

### Requisito

As respostas de erro deverão possuir estrutura uniforme contendo código, mensagem e detalhes quando aplicável.

### Justificativa

Facilita depuração e tratamento pelos consumidores da API.

---

## RNF029 — Paginação

### Requisito

Endpoints que retornem coleções deverão oferecer paginação.

### Justificativa

Evita respostas excessivamente grandes.

---

## RNF030 — Limite Máximo por Página

### Requisito

A quantidade máxima de registros por página deverá ser configurável e limitada.

### Justificativa

Evita consumo excessivo de recursos.

---

## RNF031 — Ordenação

### Requisito

Os recursos deverão permitir ordenação pelos campos previamente definidos.

### Justificativa

Melhora a flexibilidade das consultas.

---

## RNF032 — Filtros

### Requisito

Os endpoints deverão suportar filtros compatíveis com cada recurso.

### Justificativa

Reduz o volume de dados retornados.

---

## RNF033 — Pesquisa

### Requisito

Os recursos poderão oferecer pesquisa textual quando aplicável.

### Justificativa

Facilita localização de informações.

---

## RNF034 — Idempotência

### Requisito

Operações críticas deverão suportar chaves de idempotência.

### Justificativa

Evita processamento duplicado em caso de repetição de requisições.

---

## RNF035 — Compressão HTTP

### Requisito

A API deverá suportar compressão das respostas quando suportada pelo cliente.

### Justificativa

Reduz consumo de banda.

---

## RNF036 — Cache HTTP

### Requisito

Recursos apropriados deverão utilizar mecanismos de cache HTTP.

### Justificativa

Melhora desempenho e reduz carga da aplicação.

---

## RNF037 — Documentação OpenAPI

### Requisito

Toda API deverá possuir documentação baseada na especificação OpenAPI.

### Justificativa

Facilita integração e manutenção.

---

## RNF038 — Health Check

### Requisito

A aplicação deverá disponibilizar endpoints para verificação de disponibilidade e integridade dos serviços.

### Justificativa

Permite monitoramento automatizado.

---

## RNF039 — Rate Limiting

### Requisito

A API deverá limitar requisições conforme políticas configuráveis.

### Justificativa

Protege a aplicação contra abuso e ataques.

---

## RNF040 — Compatibilidade Retroativa

### Requisito

Alterações incompatíveis deverão ser disponibilizadas apenas em novas versões da API.

### Justificativa

Preserva integrações existentes e reduz impactos para consumidores.
