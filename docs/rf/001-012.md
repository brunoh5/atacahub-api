# Requisitos Funcionais (RF)

> Versão: 1.0.0
>
> Status: Em elaboração
>
> Última atualização: 03/07/2026

---

# Objetivo

Este documento descreve todos os requisitos funcionais do AtacaHub API.

Cada requisito representa uma funcionalidade que deverá ser implementada durante o desenvolvimento do sistema.

---

# Convenções

Os requisitos são identificados pelo prefixo **RF** seguido por um número sequencial.

Exemplo:

- RF001
- RF002
- RF003

Os módulos seguem uma ordem lógica de desenvolvimento, permitindo que funcionalidades posteriores dependam das anteriores.

---

# Módulo 1 — Autenticação

## Objetivo

O módulo de autenticação é responsável pela identificação dos usuários, gerenciamento de credenciais, sessões e recuperação de acesso ao sistema.

---

## RF001 — Cadastro de Usuário

### Descrição

O sistema deverá permitir o cadastro de novos usuários.

### Regras

- O e-mail deve ser único.
- O e-mail deve ser armazenado em letras minúsculas.
- A senha deverá ser armazenada utilizando algoritmo de hash.
- O usuário será criado com o status **Pendente de Verificação**.

### Dependências

Nenhuma.

### Critérios de Aceite

- O usuário é cadastrado com sucesso.
- Não é permitido cadastrar e-mails duplicados.
- A senha não é armazenada em texto puro.
- O cadastro é registrado na auditoria.

---

## RF002 — Verificação de E-mail

### Descrição

O sistema deverá permitir que o usuário confirme seu endereço de e-mail através de um token de verificação.

### Regras

- O token deve possuir tempo de expiração.
- O token poderá ser utilizado apenas uma vez.

### Dependências

RF001.

### Critérios de Aceite

- O e-mail é marcado como verificado.
- Tokens expirados são recusados.
- Tokens reutilizados são recusados.

---

## RF003 — Reenvio de Verificação

### Descrição

O sistema deverá permitir o reenvio do e-mail de verificação.

### Regras

- Apenas usuários pendentes poderão solicitar novo envio.
- O token anterior deverá ser invalidado.

### Dependências

RF001.

### Critérios de Aceite

- Um novo token é criado.
- O token anterior deixa de ser válido.

---

## RF004 — Login

### Descrição

O sistema deverá permitir que usuários autenticados acessem o sistema utilizando e-mail e senha.

### Regras

- Apenas usuários ativos poderão realizar login.
- O e-mail deverá estar verificado.

### Dependências

RF001
RF002

### Critérios de Aceite

- Um Access Token é gerado.
- Um Refresh Token é gerado.
- A sessão é registrada.

---

## RF005 — Logout

### Descrição

O sistema deverá permitir que o usuário encerre sua sessão atual.

### Regras

- O Refresh Token deverá ser invalidado.

### Dependências

RF004.

### Critérios de Aceite

- O token deixa de ser válido.
- A sessão é encerrada.

---

## RF006 — Renovação de Token

### Descrição

O sistema deverá permitir renovar um Access Token utilizando um Refresh Token válido.

### Regras

- Refresh Tokens expirados devem ser recusados.
- Refresh Tokens revogados devem ser recusados.

### Dependências

RF004.

### Critérios de Aceite

- Um novo Access Token é emitido.
- Tokens inválidos retornam erro.

---

## RF007 — Recuperação de Senha

### Descrição

O sistema deverá permitir solicitar a redefinição da senha.

### Regras

- Um token temporário deverá ser enviado por e-mail.

### Dependências

RF001.

### Critérios de Aceite

- O token é criado.
- O envio é registrado.

---

## RF008 — Redefinição de Senha

### Descrição

O sistema deverá permitir definir uma nova senha utilizando um token válido.

### Regras

- O token poderá ser utilizado apenas uma vez.
- O token deverá possuir expiração.

### Dependências

RF007.

### Critérios de Aceite

- A senha é atualizada.
- Todas as sessões existentes são encerradas.

---

## RF009 — Alteração de Senha

### Descrição

O sistema deverá permitir que usuários autenticados alterem sua senha.

### Regras

- A senha atual deverá ser informada.

### Dependências

RF004.

### Critérios de Aceite

- A senha é atualizada.
- As demais sessões são encerradas.

---

## RF010 — Gerenciamento de Sessões

### Descrição

O sistema deverá permitir visualizar e encerrar sessões ativas.

### Regras

- Cada sessão deverá armazenar dispositivo, endereço IP e data do último acesso.

### Dependências

RF004.

### Critérios de Aceite

- O usuário visualiza todas as sessões.
- É possível encerrar sessões individualmente.
- É possível encerrar todas as sessões simultaneamente.

---

## RF011 — Bloqueio por Tentativas de Login

### Descrição

O sistema deverá bloquear temporariamente novas tentativas de autenticação após sucessivas falhas de login.

### Regras

- O limite de tentativas deverá ser configurável.
- O tempo de bloqueio deverá ser configurável.

### Dependências

RF004.

### Critérios de Aceite

- Novas tentativas são recusadas durante o período de bloqueio.
- O evento é registrado nos logs.

---

## RF012 — Histórico de Acessos

### Descrição

O sistema deverá manter um histórico dos acessos realizados pelos usuários.

### Regras

- Devem ser registrados data, horário, endereço IP e dispositivo.

### Dependências

RF004.

### Critérios de Aceite

- O histórico pode ser consultado pelo próprio usuário.
- O histórico pode ser consultado por administradores.
